1. How should the Rust/WASM frontend talk to the stop data? Options:
   - Static JSON files generated by `t-pid-stops` and hosted on gh-pages
   - A small API endpoint
   - Data bundled into the WASM binary at build time

If the binary is small enough, bundle it, otherwise static files, preferably
CBOR over JSON.

2. Which mapping approach? Options:
   - Leaflet (JS) with WASM calling into it via wasm-bindgen/js-sys
   - A pure-Rust tile renderer (more work)

No preference, do whatever is less error-prone.

3. Should the gh-pages deployment be automatic on merge to main, or manual?

Automatic on merge, automatic update nightly.

4. What's the expected number of stops? This affects whether we can render all markers at once
   or need clustering.

Download the file and find out but I guess we need clustering.
